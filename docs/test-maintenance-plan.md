# Monorepo 测试维护计划（集中到 `test/` 目录）

> 目标：在“统一测试目录 + 高速迭代”的前提下，实现可持续、可观测、可回归的质量体系。

## 1. 背景与目标

当前仓库是 monorepo，包含接口层、脚本层、业务逻辑层与功能函数层。统一收敛测试到 `test/` 的价值在于：

1. **治理统一**：命名、分层、运行入口、报告格式一致。
2. **认知统一**：新人可快速定位测试资产与质量标准。
3. **CI 统一**：变更影响分析、分层阻断、回归基线都能在同一流水线实现。

核心目标：

- 保障迭代速度：PR 能快速得到“该变更是否破坏关键能力”的反馈。
- 保障稳定性：线上关键路径（接口、核心业务规则、脚本发布链路）可持续回归。

---

## 2. 测试目录与分层规范（统一到 `test/`）

建议采用“按层 + 按领域”双维组织：

```txt
test/
  unit/                  # 功能函数、纯业务逻辑（快）
    domain/
    utils/
  integration/           # 跨模块集成：接口到服务、脚本到模块
    api/
    mcp/
    scripts/
  e2e/                   # 关键业务流程端到端（慢）
    gateway/
    release/
  contract/              # 接口契约/manifest/schema 一致性
  fixtures/              # 测试数据样本（csv/json/yaml 等）
  helpers/               # 测试工具函数（mock 工厂、builder）
  smoke/                 # 发布前冒烟（核心路径）
```

命名规范：

- 文件统一：`*.test.ts`
- 场景命名：`<module>.<layer>.test.ts`（如 `bom-pricing.unit.test.ts`）
- Fixtures：`test/fixtures/<domain>/<case-name>.*`

---

## 3. 覆盖策略：接口到脚本，从业务逻辑到功能函数

### 3.1 功能函数（unit）

**对象**：纯函数、转换器、校验器、算法函数。

**要求**：

- 正常输入、边界输入、异常输入全覆盖。
- 关键算法使用“表驱动测试”（输入/输出矩阵）。
- 对外部 IO（时间、随机数、文件）全部 mock。

### 3.2 业务逻辑（unit + integration）

**对象**：领域服务、规则引擎、状态流转。

**要求**：

- 单测验证“规则正确性”。
- 集成测试验证“模块协同与状态一致性”。
- 每个核心业务流程至少一个“黄金路径 + 一个失败路径”。

### 3.3 接口层（integration + contract）

**对象**：MCP 工具、HTTP/gateway、配置协议。

**要求**：

- 契约测试锁定输入输出 schema（字段、类型、必填、默认值）。
- 错误码与错误消息结构稳定（避免前端/调用方回归故障）。
- 版本兼容测试（新增字段不破坏旧调用方）。

### 3.4 脚本层（integration + smoke）

**对象**：发布脚本、构建脚本、打包脚本、迁移脚本。

**要求**：

- 对“参数解析、路径处理、产物命名、失败退出码”做集成测试。
- 发布前 smoke 覆盖最小闭环（例如：生成产物 + 校验 manifest）。
- 对高风险脚本增加 dry-run 模式测试。

---

## 4. 质量门禁（兼顾迭代与稳定）

采用分级门禁，避免“所有测试都阻断”导致研发效率下降。

### 4.1 PR 阶段（快速反馈）

- 必跑：`unit + contract + 变更影响的 integration 子集`
- 时长目标：10 分钟以内
- 阻断条件：
  - 新增/修改代码对应测试缺失
  - 核心模块覆盖率低于阈值
  - 契约测试失败

### 4.2 主干合并后（完整回归）

- 运行：完整 `integration + smoke`
- 每日定时：`e2e` 全量回归
- 输出：失败趋势、慢测榜单、波动用例名单

### 4.3 发布前（稳定优先）

- 必过：`smoke + 关键 e2e + 发布脚本集成测试`
- 强制冻结：关键失败未修复不得发布

---

## 5. 变更影响分析（Monorepo 必备）

为避免每次全量跑，建议建立“模块 -> 测试集”映射表：

- `mcp/bom-mcp/**` 变更：触发 `test/unit/domain + test/integration/mcp + test/contract`
- `scripts/**` 变更：触发 `test/integration/scripts + test/smoke`
- `gateway/protocol` 变更：触发 `test/integration/api + test/e2e/gateway + test/contract`

实现方式：

1. 读取 git diff 文件列表。
2. 根据映射规则选择最小测试集。
3. PR 仅跑最小集，夜间跑全量。

---

## 6. 稳定性策略：防止“测试本身不稳定”

1. **隔离外部依赖**：网络、时间、随机数、文件系统尽量可注入。
2. **固定测试数据**：fixture 版本化，禁止线上动态数据直连。
3. **并发控制**：共享资源（端口、临时目录）加唯一前缀。
4. **Flaky 治理**：
   - 连续 N 次失败自动打标；
   - 24h 内必须归因；
   - 未修复用例隔离到 quarantine，且需负责人。

---

## 7. 覆盖率与质量指标（KPI）

建议以“关键路径覆盖”替代“盲目总覆盖率”。

- 核心业务模块（domain）：行覆盖率 ≥ 85%
- 工具函数模块（utils）：行覆盖率 ≥ 90%
- 接口契约模块：关键契约 100% 覆盖
- 发布脚本：关键分支覆盖（成功、参数错误、IO 失败）

质量运营指标：

- PR 平均反馈时长
- 回归失败率（按层统计）
- flaky 用例数量与修复时长
- 漏测导致线上回滚次数

---

## 8. 组织与职责

- **模块 Owner**：对本模块测试完整性负责。
- **QA/质量 Owner**：维护测试基线、门禁规则、周报。
- **发布 Owner**：对 smoke 与发布脚本测试结果负责。

流程建议：

1. 新功能 PR 必带测试（至少 unit 或 contract）。
2. 缺陷修复 PR 必带“复现用例 + 防回归用例”。
3. 每周 30 分钟质量例会：审查 flaky、慢测、漏测点。

---

## 9. 三阶段落地路线图

### 阶段 A（1-2 周）：统一与可运行

- 完成 `test/` 分层目录迁移。
- 统一命名、fixtures、helpers。
- 建立一条标准 `bun test` 入口。

### 阶段 B（2-4 周）：门禁与影响分析

- 接入 PR 分层门禁。
- 完成变更影响分析脚本（按路径选择测试集）。
- 建立 contract 测试基线。

### 阶段 C（持续）：稳定性运营

- 形成 flaky 台账与修复 SLA。
- 引入 nightly 全量回归报告。
- 每月复盘“漏测导致事故”，持续补盲。

---

## 10. DoD（计划完成定义）

满足以下条件，说明“测试集中化 + 稳定迭代”机制落地：

1. `test/` 目录成为唯一测试入口，旧分散测试清理完毕。
2. PR 能在 10 分钟内给出分层反馈（至少 unit/contract）。
3. 发布前 smoke 与关键 e2e 稳定可复现。
4. flaky 用例有记录、有负责人、有关闭时限。
5. 关键模块覆盖率达到目标且持续可视化。
