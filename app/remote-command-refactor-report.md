# ClawOS 远程指令编排改造分析报告

## 1. 背景与目标

你提出的新执行链路是：

> 点击按钮 → 提交环境和动作信息到云端 → 云端返回本地逻辑指令组合 → 本地执行

核心目标：

- 把“动作实现”从客户端硬编码迁移为“云端可编排指令”。
- 后续新增/调整动作时，尽量不发客户端版本，只调整云端策略与指令模板。
- 保留当前本地任务执行、日志中心、状态轮询等能力，减少重构风险。

---

## 2. 当前架构现状（与改造相关）

当前 App 的动作大多是本地固定路由：

- 前端按钮直接调用本地 API（如 `startBrowserAction`、`startGatewayAction`、`startEnvironmentInstall` 等）。
- 本地服务端按 action 类型创建本地任务，并调用固定实现。
- 任务日志在本地维护，UI 轮询本地任务状态。

问题点：

1. **动作语义与执行细节耦合在客户端**
   - 新增动作/变更步骤需要改客户端代码并发布。
2. **跨版本兼容成本高**
   - 不同版本客户端动作参数可能分叉，维护困难。
3. **策略不可热更新**
   - 例如“安装顺序、重试策略、条件分支”无法快速在线调整。

---

## 3. 目标方案（建议）

## 3.1 逻辑分层

引入“远程动作编排层（Cloud Planner）+ 本地指令运行时（Local Runtime）”：

- **Cloud Planner（云端）**
  - 输入：`actionIntent + envSnapshot + clientCaps`
  - 输出：`commandPlan`（可执行命令组合）
- **Local Runtime（客户端）**
  - 负责校验、执行、回传日志与结果
  - 保留本地安全策略与白名单

## 3.2 新执行链路

1. 用户在 UI 点击动作按钮。  
2. 客户端收集上下文：
   - 操作系统/WSL 状态
   - 工具版本（python/uv/bun 等）
   - 网关与配置状态
   - 客户端版本、能力标识（capabilities）
3. 客户端向云端提交 `plan request`。  
4. 云端返回 `commandPlan`（步骤数组 + 条件 + 回滚 + 重试策略）。  
5. 本地运行时进行：
   - schema 校验
   - 签名校验（可选但强烈建议）
   - 白名单与风险检查
6. 本地按计划执行并上报进度，日志继续进入现有日志中心。  
7. 执行完成后回传结果，云端沉淀策略效果数据。

---

## 4. 数据契约建议

## 4.1 Plan Request（客户端 -> 云端）

建议字段：

- `requestId`
- `actionIntent`（如 `gateway.restart`, `browser.open_cdp`）
- `clientVersion`
- `capabilities`（支持的执行器/命令族）
- `envSnapshot`
  - os/wsl
  - installed tools + versions
  - network 基础状态
- `userContext`（租户、组织、灰度标签）

## 4.2 Command Plan（云端 -> 客户端）

建议字段：

- `planId` / `traceId`
- `version`（计划格式版本）
- `ttlSeconds`（防重放）
- `steps[]`
  - `id`
  - `executor`（仅允许：`shell` / `powershell` / `wsl`）
  - `args`
  - `timeoutMs`
  - `retryPolicy`
  - `onFailure`（continue/abort/rollback）
  - `artifacts`（需要采集的输出）
- `constraints`
  - `allowedExecutors`: ["shell", "powershell", "wsl"]（固定三种，不可动态扩展）
  - 允许命令白名单标签
  - 权限边界
- `signature`（用于防篡改）

## 4.3 执行回传（客户端 -> 云端）

- step 级别开始/结束/耗时/退出码
- stdout/stderr 摘要（脱敏）
- 最终状态 + 结构化错误码

---

## 5. 客户端需要的调整（落地项）

## 5.1 新增模块

1. **Plan Client**
   - 负责向云端请求 `commandPlan`
2. **Plan Validator**
   - 本地 schema + 签名 + TTL 校验
3. **Command Runtime**
   - 把 `steps` 映射到现有 task runner
4. **Policy Guard**
   - 命令白名单、路径白名单、危险参数拦截

## 5.2 现有模块改造

1. 现有 `startXXXAction` 路径：
   - 从“直接执行本地逻辑”改为“请求云端计划 + 执行计划”。
2. 任务中心：
   - 扩展 step 级日志与 planId 展示。
3. 错误模型：
   - 增加远程编排错误码（拉取失败、签名非法、计划过期、策略拒绝）。

## 5.3 降级策略（必须）

- 云端不可用时可配置为：
  - **严格模式**：直接失败（高安全）
  - **兼容模式**：回落到本地旧逻辑（高可用）

建议先上兼容模式做灰度，再逐步切严格模式。

---

## 6. 云端需要的能力

1. **Action Planner 服务**：根据环境生成计划。  
2. **模板与策略中心**：动作模板版本化、灰度发布。  
3. **签名服务**：对计划签名。  
4. **观测平台**：按 planId 统计成功率、耗时、失败热点。  
5. **审计能力**：谁发布了什么策略、影响哪些租户。

---

## 7. 工作量评估（粗估）

> 以“已有本地任务系统可复用”为前提，按 1 名主程 + 1 名后端 + 1 名前端协作估算。

## 阶段 A：架构与契约（1~1.5 周）

- 统一 actionIntent 命名
- 定义 request/plan/result schema
- 定义签名与安全边界

**产出**：RFC + JSON Schema + 联调 mock

## 阶段 B：客户端 Runtime 接入（2~3 周）

- 新增 Plan Client / Validator / Runtime / Guard
- 改造 2~3 个核心动作试点（例如：重启网关、浏览器打开 CDP、环境安装）
- 日志中心增加 step 粒度展示

**产出**：可灰度的最小闭环

## 阶段 C：云端 Planner 服务（2~3 周）

- 计划生成、模板版本化、签名
- 基础观测与审计

**产出**：可配置策略后台 + API

## 阶段 D：扩面与灰度（1.5~2.5 周）

- 覆盖主要按钮动作
- 分租户/分版本灰度
- 回归与压测

**总计**：约 **6.5 ~ 10 周**（取决于云端基础设施现状）

---

## 8. 可能存在的问题与风险

1. **安全风险（最高优先级）**
   - 云端下发命令本质是“远程代码执行能力”，必须有：
     - 强签名
     - 白名单执行器
     - 高危命令封禁
     - 计划 TTL + nonce 防重放

2. **环境漂移导致计划失效**
   - 计划生成时环境是 A，执行时变成 B。
   - 需在本地执行前做 preflight 检查，不满足则拒绝执行并回传。

3. **云端可用性影响本地动作可用性**
   - 需要本地缓存最近成功计划（短期缓存）+ 降级模式。

4. **调试复杂度上升**
   - 问题可能发生在 UI、本地 runtime、云 planner 任一层。
   - 必须贯通 traceId 与结构化日志。

5. **版本兼容问题**
   - 新 plan 字段可能不被旧客户端识别。
   - 通过 `capabilities + schemaVersion` 做协商。

6. **成本与治理问题**
   - 策略发布等同“线上变更”，需要审批、审计、回滚机制。

---

## 9. 建议的实施顺序（降低风险）

1. 先做 **“双轨制”**：
   - 默认本地逻辑
   - 配置开关启用远程计划
2. 先接入 **低风险动作**（查询/检测类）
3. 再接入 **中风险动作**（重启、更新）
4. 最后接入 **高风险动作**（安装、系统修复）
5. 每类动作都要求：
   - 可观测
   - 可回滚
   - 可开关

---

## 10. 结论

这个方向是可行的，而且非常适合你要的“客户端少发版、指令逻辑可热更新”的目标。  
但它不是简单接口改造，而是一次 **执行模型升级**：从“本地硬编码动作”升级到“云端编排 + 本地受控执行”。

如果按上面的分阶段方案推进，建议第一期只覆盖 2~3 个核心按钮，先验证：

- 安全边界可控
- 云端策略可稳定生效
- 日志与排障链路完整

验证通过后再全面迁移其余动作。

## 11. 补充：将 ClawOS 桌面交互纳入远程指令

你补充的点非常关键：除了环境安装、网关、MCP 等动作，**App 桌面本身的交互**也可以纳入同一套远程编排。

建议新增一类 `actionIntent`：`app.interaction.*`，典型包括：

- `app.interaction.upgrade.start`：触发升级流程
- `app.interaction.restart`：触发重启
- `app.interaction.log_center.open`：打开日志中心
- `app.interaction.toast`：展示状态提示/错误提示
- `app.interaction.window.focus`：聚焦主窗口

这样可以让“交互行为”与“系统动作”共用同一套：

- 云端策略编排
- 本地受控执行
- 统一日志与审计

### 11.1 交互类指令约束建议

- 仍然只允许三种 executor：`shell` / `powershell` / `wsl`；App 交互通过这三类执行器触发客户端已注册的受控动作映射，不新增 executor 类型。
- 涉及用户感知的高风险交互（如重启、升级）建议支持 `requireUserConfirm` 标记。
- 所有交互类指令都要记录 `traceId + operator + result`，便于审计与回溯。

### 11.2 最小落地方式

第一期可先把以下 3 个交互接到远程计划：

1. 升级（`app.interaction.upgrade.start`）
2. 重启（`app.interaction.restart`）
3. 打开日志中心（`app.interaction.log_center.open`）

这样能快速验证：

- 云端下发交互策略是否稳定
- 本地 UI 编排与任务系统是否兼容
- 安全开关（确认弹窗/白名单）是否满足预期
