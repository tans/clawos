# ClawOS 行业分支定制方案（从商业服务到技术落地）

## 1. 定位与目标

ClawOS 基于 Windows + Bun + WSL + openclaw Gateway，天然适合做“行业化本地智能工作台”。
本方案目标是把“按行业定制”做成可规模化复用的业务，而不是一次性项目外包。

核心目标：

1. 商业上：形成标准化服务包 + 行业增值包 + 持续运维收入。
2. 产品上：保持主干统一，行业差异通过分支与配置插件化管理。
3. 交付上：做到 2-6 周可上线一个行业版本，控制成本与质量风险。

## 2. 商业服务设计

## 2.1 产品与服务分层

1. 基础版（通用）
   - 控制面板、渠道配置、Agent 配置、技能配置、浏览器配置、自启动。
   - 适合中小客户快速接入。

2. 行业版（教育/医疗/制造等）
   - 行业默认模板（提示词、技能、流程、权限、审计字段）。
   - 行业集成适配（常见系统接口、报表字段、术语词典）。

3. 企业版（深度定制）
   - 私有部署、SSO、审计增强、合规策略、专属插件。
   - 专项实施与 SLA。

## 2.2 收费模式建议

1. 一次性实施费
   - 需求梳理、原型、接口对接、试点上线。

2. 年度订阅费
   - 版本更新、行业规则更新、安全补丁、技术支持。

3. 增值收费
   - 新系统对接、专属模型策略、报表与运营看板、培训认证。

## 2.3 交付形态

1. 标准安装包（Windows exe）+ 行业配置包（`clawos_xiake.json` 的行业变体）。
2. 官网下载中心分行业入口（教育版、医疗版）。
3. 上传发布流程：通过发布接口上传安装包和配置，自动更新 latest。

## 3. 行业方案模板（教育/医疗示例）

## 3.1 教育行业（Edu）

业务场景：

1. 招生咨询与问答辅助。
2. 教务通知与自动化信息流转。
3. 课程资料检索与知识问答。

行业定制点：

1. 术语词典：课程、学段、班级、考试节点。
2. 模板技能：招生 FAQ、通知生成、家校沟通标准话术。
3. 权限与审计：按校区/部门隔离，保留问答审计日志。

集成优先级：

1. 企微（优先）。
2. 内部教务系统接口（第二阶段）。

## 3.2 医疗行业（Med）

业务场景：

1. 导诊与院内流程指引（非诊断）。
2. 医务行政文档整理与模板生成。
3. 知识库检索（规范、流程、制度）。

行业定制点：

1. 合规限制：敏感信息脱敏、最小权限、审计留痕。
2. 输出边界：明确“仅辅助，不替代医疗判断”。
3. 风险提示：高风险问题自动转人工流程。

集成优先级：

1. 企微/内部 IM。
2. HIS/EMR 仅读接口（从只读开始，逐步扩展）。

## 4. 从商机到上线的交付流程

## 4.1 商务阶段（1-2 周）

1. 线索评估：行业、规模、系统复杂度、合规等级。
2. 方案演示：通用版 + 行业样板。
3. 合同范围：明确“标准能力 / 定制能力 / 排除项”。

输出物：

1. 《需求澄清表》
2. 《实施边界清单》
3. 《里程碑计划与报价》

## 4.2 实施阶段（2-6 周）

1. 第 1 周：需求冻结与系统盘点。
2. 第 2-3 周：行业配置与技能定制、接口联调。
3. 第 4 周：UAT、培训、上线演练。
4. 扩展周：高优先级增补需求。

输出物：

1. 行业分支代码/配置
2. 安装包与配置包
3. 操作手册、运维手册、验收报告

## 4.3 运营阶段（持续）

1. 月度回顾：使用率、成功率、人工转接率、错误率。
2. 版本节奏：双周修复、月度功能更新、季度行业升级。
3. SLA：故障分级响应（P1/P2/P3）。

## 5. 技术落地方案（Branch + 配置驱动）

## 5.1 代码与分支策略

1. 主干 `main`：公共能力，必须稳定。
2. 行业分支：`industry/edu`、`industry/med`。
3. 客户分支：`customer/<industry>/<name>`（仅在确需专属逻辑时创建）。

原则：

1. 先配置，后代码；先插件，后分叉。
2. 行业通用能力回流主干，避免分支长期漂移。

## 5.2 配置分层

建议目录：

```txt
configs/
  base/
    clawos.base.json
  industry/
    edu.json
    med.json
  customer/
    edu-school-a.json
    med-hospital-b.json
```

加载优先级：

1. `base`
2. `industry`
3. `customer`
4. 本地覆盖（现场应急）

## 5.3 插件化能力边界

可插件化模块：

1. 行业技能包（skills）
2. 渠道适配（channel adapters）
3. 审计策略（日志字段、敏感词规则、脱敏器）
4. 行业仪表盘（统计口径）

不建议频繁定制模块：

1. 更新流程核心逻辑（WSL 更新流水线）
2. 任务执行器基础框架
3. 网关连接底层协议实现

## 5.4 发布与分发流程

结合现有 web 发布能力：

1. CI 打包：生成行业安装包（如 `clawos-edu-1.3.0.exe`）。
2. 上传配置包：`clawos_xiake.edu.json` / `clawos_xiake.med.json`。
3. 上传安装包：调用上传接口更新 latest 元数据。
4. 官网下载：按行业展示下载入口和版本说明。

## 6. 组织与角色分工

1. 行业顾问（业务 Owner）
   - 负责场景梳理、术语定义、验收标准。
2. 产品经理
   - 管理行业模板、版本路线与需求优先级。
3. 实施工程师
   - 负责现场部署、联调、培训、上线保障。
4. 平台工程师
   - 负责主干稳定、发布系统、安全合规能力。

## 7. 关键指标（KPI）

商业指标：

1. 行业版签约转化率
2. 单客户实施毛利率
3. 年续费率

交付指标：

1. 首次上线周期（TTV）
2. UAT 一次通过率
3. 线上 P1 故障数

产品指标：

1. 行业模板复用率
2. 分支回流主干比例
3. 每版本回归缺陷率

## 8. 风险与控制

1. 分支失控
   - 控制：严格分层与回流机制；禁止在客户分支堆积公共功能。

2. 合规风险（医疗尤其明显）
   - 控制：最小权限、脱敏、审计、输出边界提示。

3. 需求蔓延
   - 控制：合同中定义“标准项/定制项/变更流程”。

4. 发布混乱
   - 控制：统一上传发布入口，latest 元数据自动化更新。

## 9. 90 天落地计划（建议）

第 1-30 天：

1. 完成行业模板 v1（教育、医疗）。
2. 建立分支规范、配置分层规范、发布规范。
3. 完成官网行业下载页与上传发布流程联动。

第 31-60 天：

1. 教育行业首个试点上线。
2. 医疗行业 PoC 完成（含合规基线）。
3. 建立实施手册与培训材料。

第 61-90 天：

1. 形成可复制交付包（售前演示 + 快速实施）。
2. 完成 2-3 个标杆案例。
3. 复盘并回流行业共性能力到主干。

## 10. 结论

ClawOS 的行业化策略应采用“主干稳定 + 分支可控 + 配置驱动 + 发布统一”的方式。
这样可以同时满足教育、医疗等行业的差异化需求，并保持长期商业化与技术可持续性。
